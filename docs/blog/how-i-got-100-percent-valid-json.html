How I got 100% valid JSON from local LLMs with zero retries

Fatih Dinc · April 2026 · 8 min read

If you've tried to get structured data out of an LLM, you know the pattern: ask for JSON, parse the response, catch the exception, retry. Maybe add "ONLY output valid JSON" to the prompt. Maybe wrap the output in a regex. Maybe try three times and hope one works.

I got tired of hoping. I wanted a guarantee.

Not "99.7% valid." Not "usually works after one retry." Actual, mathematical, provable 100% for the schema constructs I support. And I wanted it running locally on my MacBook, not through an API that costs money per token and sees all my data.

This is how I built it.

The problem is structural

LLMs generate text one token at a time. Each token is selected from a probability distribution over tens of thousands of vocabulary entries. The model picks the most likely next token, appends it, and repeats.

Nothing in this process knows about JSON. The model has seen JSON in training data, so it has a statistical tendency to produce valid syntax. But tendencies aren't guarantees. A model might emit {"name": "Alice", "age": twenty-nine} because "twenty-nine" is a plausible continuation of "age":

Post-hoc validation catches the error after it happens. Retrying hopes the model makes different choices the second time. Neither approach eliminates the root cause: the model can choose tokens that break the structure.

The fix: don't let it choose wrong

The core idea is simple. Before the model picks a token, figure out which tokens would produce valid JSON. Mask everything else. The model can only choose from valid continuations.

This is called constrained decoding. The concept isn't new. XGrammar, Outlines, and Guidance all implement variations. But they target CUDA and CPU runtimes. I wanted something for Apple Silicon, using MLX for direct GPU access.

The implementation has three components.

Component 1: VocabScanner

Every model has a vocabulary: a fixed set of tokens it can produce. Qwen2.5-14B has about 150,000 tokens. Llama-3.1-8B has about 128,000.

VocabScanner classifies each token in the vocabulary by its structural role in JSON. It decodes each token to text and pattern-matches it:

  • { is LBRACE, } is RBRACE
  • "some text" is STRING
  • 42 or 3.14 is NUMBER
  • true, false, null are literals
  • : is COLON, , is COMMA

This classification runs once per tokenizer and gets cached to disk. Second run is instant.

# Conceptually:
for token_id in range(vocab_size):
    text = tokenizer.decode([token_id])
    token_type = classify(text)  # LBRACE, STRING, NUMBER, ...
    mapping[token_id] = token_type

Component 2: JSONContextTracker

JSON has grammar. After { you need a string key or }. After a key you need :. After a value you need , or a closing bracket.

JSONContextTracker is a stack-based state machine that follows the JSON being generated character by character. At any point during generation, it knows exactly which token types are valid next.

When the schema says {"type": "object", "properties": {"name": {"type": "string"}}}, the tracker knows:

  • Step 0: valid = {LBRACE}. Must open an object.
  • Step 1: valid = {STRING_KEY}. Must produce a key.
  • Step 2: valid = {COLON}. Must separate key from value.
  • Step 3: valid = {STRING}. The value must be a string.
  • Step 4: valid = {COMMA, RBRACE}. More fields or close.

The stack handles nesting. Objects inside objects, arrays inside objects, arbitrary depth. The tracker pushes a frame when entering a nested structure and pops when leaving.

Component 3: AND Gate

This is where it comes together. At each generation step:

  1. The model produces a logit vector: one score per vocabulary token
  2. The tracker says which token types are valid
  3. The AND Gate sets the logit of every invalid token to negative infinity
  4. The model picks from the remaining tokens
# The AND Gate in three lines:
mask = [token_type[i] in valid_types for i in range(vocab_size)]
logits[~mask] = -inf
next_token = argmax(logits)

After softmax, tokens with -infinity logits have zero probability. They cannot be selected. The model's creative freedom is preserved within the valid set, but it physically cannot produce a token that would break the JSON structure.

The guarantee: If every token is constrained to be structurally valid, the complete output is structurally valid. This isn't a statistical claim. It's a logical consequence of the mechanism.

Does it actually work?

I ran 80 benchmark trials across two models (Qwen2.5-14B-Instruct-4bit and Llama-3.1-8B-Instruct-4bit) and multiple schema types (flat objects, nested objects, arrays, anyOf unions). Every trial produced valid, schema-conformant JSON.

80 out of 80. 100%. Zero retries on any trial.

The 95% confidence interval for the true success rate is [95.5%, 100%], using the Clopper-Pearson exact method. With 80 independent successes and zero failures, the lower bound is already above 95%.

What it looks like in practice

from llmjson import generate

result = generate(
    model_id="mlx-community/Qwen2.5-14B-Instruct-4bit",
    prompt="Generate a person profile for Alice, a senior engineer.",
    schema={
        "type": "object",
        "properties": {
            "name": {"type": "string"},
            "age": {"type": "integer"},
            "email": {"type": "string"},
        }
    },
)

print(result.text)
# {"name": "Alice Chen", "age": 32, "email": "alice.chen@eng.dev"}

print(result.is_valid)
# True

Or from the command line:

$ llmjson generate \
    --model mlx-community/Qwen2.5-14B-Instruct-4bit \
    --schema person.schema.json \
    --prompt "Profile for Alice"

{"name": "Alice Chen", "age": 32, "email": "alice@example.com", "active": true}

Limitations (honest ones)

This approach has real constraints. Ignoring them would be dishonest.

Apple Silicon only. llmjson uses MLX for direct logit access. If you're on Linux with NVIDIA, use Outlines or XGrammar. They're good tools built for that hardware.

Schema subset. The tracker supports JSON Schema's common constructs: objects, arrays, primitives, nested structures, anyOf. It does not support $ref, patternProperties, additionalProperties, conditional schemas (if/then/else), or regex patterns. The validator rejects unsupported schemas upfront, so you find out immediately instead of getting wrong output.

Content quality is the model's job. Constrained decoding guarantees the structure. Whether the model puts sensible values in the fields is up to the model. A bad prompt with a good schema still gives you valid JSON with mediocre content.

Token alignment. Most tokenizers split text into subword tokens that don't align perfectly with JSON structural characters. A string like "name": might be one token or four. The tracker handles this by processing decoded text character by character, but edge cases exist with unusual tokenizers.

Ollama does this too. I ran the same 4 schemas through Ollama's --format json with Qwen2.5:14b. It scored 40/40. Ollama uses GBNF grammars to constrain output and it works. If you already use Ollama and don't care about speed, it's a fine choice. llmjson is 10-25x faster on the same hardware because it talks to MLX directly instead of going through a server.

Why llmjson over Ollama?

Both guarantee valid JSON. The differences are practical:

  • Speed: llmjson generates in ~1-2 seconds. Ollama takes 10-25 seconds for the same schema on the same Mac. Direct MLX access vs. HTTP intermediary.
  • No daemon: llmjson is a Python library. Import it, call it. No background server to manage.
  • Privacy: The data never leaves your process. No local network socket, no shared memory.
  • Logit access: You get the raw probability distribution. Useful if you're building something beyond basic JSON generation.

If you need an HTTP API, use Ollama. If you want a library you can embed in your Python code with maximum speed, use llmjson.

How to try it

pip install llmjson

You'll need Python 3.9+ and an Apple Silicon Mac (M1 or later). The MLX dependencies install automatically.

The first run downloads the model (~8 GB for Qwen2.5-14B-4bit) and builds the token classification cache. Subsequent runs are fast.
